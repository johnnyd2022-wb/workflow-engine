1. The correct model

You have:

Categories (what type of problem)
Expiry / validity (raw materials, outputs)
Process timing (ready-before-use)
Traceability (untracked items)

These are labels, not severity.

Events (what actually happened)

Severity is only determined at the event level:

“Was this rule violated in active production?”

2. So are both categories critical?
❌ Incorrect framing
“Is expiry category critical?”
“Is traceability category critical?”
✅ Correct framing
“Was an expired item used in production?”
“Was an untracked item used in production?”
3. Final answer in your terms
Expired raw materials / expired outputs
Critical WHEN USED
Otherwise not critical
Ready items used before ready date
Critical WHEN USED
Otherwise not critical
Untracked items
Critical WHEN USED
Otherwise not critical
4. So what about traceability?

This is the only nuance:

Traceability is not “less critical”

It is just a different dimension of failure:

Expiry / readiness → product integrity failure
Untracked → audit / traceability failure

But both can be:

CRITICAL when they affect active production

5. Clean rule you can rely on
A single rule governs everything:

If a constraint is violated during active production → CRITICAL event

6. What this means for your system status bar

You should NOT have:

“critical inventory issues”
“critical traceability issues”

Instead:

System Health
CRITICAL if ANY:
expired item used
unready item used
untracked item used
7. If you want one extra improvement (important)

Split internally (not in UI severity):

Event types:
QUALITY_BREACH → expiry / readiness
TRACEABILITY_BREACH → untracked
Severity:
derived only from:
whether production was impacted
8. Final clarity statement

Categories are never critical.
Only violations in active production are critical.
