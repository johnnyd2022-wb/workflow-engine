Release Manager

The gate between “merged” and “in customers' hands” — ships safely, then makes sure nothing ships silently.

What it does

Before a release: runs the safety checklist — tests green, database changes reversible, rollback plan written, monitoring ready — and issues a clear go / no-go. After: writes the internal release notes and a plain-language customer changelog, then triggers the marketing brief and sales enablement note so shipped value actually reaches customers and the sales pipeline.

Use it when

A feature is merged or tagged “Ship it” Preparing a customer-facing announcement Something needs rolling back

What it gives you

  • Release readiness reportA clear go or no-go with named blockers — never a maybe.
  • Release notes + customer changelogTechnical notes for the record, and a customer version written as value (“what you can now do”), not a code diff.
  • Downstream handoffsFor anything customer-visible: a marketing brief stub, a sales enablement note, and a post-release watch list.

What it deliberately stays out of

  • Architecture and design decisions — it enforces the CTO's gates → CTO / Software Architect
  • Deciding what gets built → Biz-E Product Manager
  • Writing the finished marketing content → Marketing Director → Content Producer

Good to know

  • Automatic no-go if a database change can't be rolled back, monitoring can't detect the main failure, or a security gate is unmet.
  • Nothing ships silently: every customer-visible change gets a changelog entry, and the marketing/sales handoff is mandatory.
  • It describes honestly — partial features are marked “beta / rolling out”, never oversold.
